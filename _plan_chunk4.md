## Root Causes

1. **Source-string-as-test pattern**: These tests were likely generated by AI or written to prevent regressions from specific refactoring runs. They test "the source code contains this string" rather than "this behavior produces this result." This makes them brittle barriers to future refactoring.
2. **Regression directory proliferation**: The regression/ taxonomy (analysis, cross_cutting, report, runtime) was created to organize regression tests by domain, but no enforcement prevented cross-domain bleeding. Files were added to whichever directory seemed closest.
3. **Profile wrapper proliferation**: Each `make_profile_*` function adds a car profile to base fault samples, but they're all thin wrappers with the same pattern. A single parameterized function would suffice.

## Simplification Approach

### Step 1: Replace inspect.getsource tests with behavioral equivalents

For each of the 31 `inspect.getsource()` tests, determine what behavior they're guarding and write a direct behavioral test. Categories:

**Category A — "Code contains string X"**: Tests like `assert "min(confidence, _NEGLIGIBLE_STRENGTH_CONF_CAP)" in source`. Replace with: call the function with inputs that exercise the cap, assert output is capped.

**Category B — "Code does NOT contain string X"**: Tests like `assert "_owns_pool" not in source`. Replace with: test the function's behavior to confirm the pool is not owned (or delete if the test is for a specific past bug that's now irrelevant).

**Category C — "Lock usage in source"**: Tests like `assert "self._lock" in source`. Replace with: behavioral concurrency test or delete if the lock is an implementation detail.

**Category D — "Exception handling in source"**: Tests like `assert "raise ValueError" in source`. Replace with: call the function with invalid input, assert ValueError is raised.

Process per file:
1. Read each test function
2. Determine the guarded behavior
3. Write a behavioral replacement (or delete if the guard is for a one-time migration)
4. Remove `import inspect` once all `getsource` calls are gone from the file
5. If a file becomes empty after removing all source-inspection tests, delete the file

### Step 2: Consolidate regression/ directory structure

Current: 4 subdirectories (analysis/8, cross_cutting/7, runtime/10, report/3) = 29 files
Target: Flatten to 2 subdirectories:
- `regression/analysis/` — tests for analysis pipeline behavior
- `regression/runtime/` — tests for runtime, persistence, API, WS behavior
- `regression/report/` — tests for report rendering behavior

Moves:
- `cross_cutting/test_report_delivery_and_stream_resilience_regressions.py` → split: report parts to report/, runtime parts to runtime/
- `cross_cutting/test_coverage_gap_audit_round2.py` → merge with `analysis/test_coverage_gap_audit_top10.py`
- `cross_cutting/test_review_guardrails_*` (2 files) — these are primarily source-string tests; will be mostly deleted in Step 1, residual behavioral tests go to analysis/ or runtime/
- `cross_cutting/test_exception_discipline.py` → runtime/ (cross-module exception handling)
- `cross_cutting/test_refactor_contract_regressions.py` → runtime/ (contract preservation)
- `cross_cutting/test_multi_domain_regressions.py` → runtime/ (multi-subsystem regressions)

After Step 1 removes source-inspection tests, many files will shrink or become empty, naturally reducing the count.

### Step 3: Consolidate profile wrapper builders

In `test_support/fault_scenarios.py`, replace the 6 `make_profile_*` functions with a single parameterized helper:

```python
def make_fault_samples(
    fault_type: str,
    *,
    car_profile: CarProfile | None = None,
    speed_sweep: bool = False,
    ...
) -> list[SensorFrame]:
```

This eliminates `make_profile_fault_samples`, `make_profile_dual_fault_samples`, `make_profile_engine_order_samples`, `make_profile_speed_sweep_fault_samples`, `make_profile_gain_mismatch_samples`, and `build_fault_samples_at_speed` as separate functions.

Update all callers to use the unified function with keyword arguments.

### Step 4: Eliminate secondary sample path in report_analysis_integration.py

`report_analysis_integration.py` imports `_make_sample` from `report_helpers.py` and renames it `analysis_sample`. This creates a confusing secondary sample-creation path. Replace with direct use of `make_sample` from `sample_scenarios.py` with appropriate parameters.

## Risks and Tradeoffs

1. **Behavioral test conversion**: Some source-inspection tests guard against subtle bugs. Converting to behavioral tests requires understanding the original intent. Risk: missing the exact edge case. Mitigated: examine each test's commit history/comments for context.
2. **Test deletion**: Deleting source-inspection tests without replacement reduces test count. Risk: uncovered regression. Mitigated: only delete tests whose guarded behavior is already covered by other tests or is for one-time migrations.
3. **Builder consolidation**: Merging `make_profile_*` functions changes the test API. Risk: callers may pass wrong parameters. Mitigated: keyword-only arguments prevent positional mistakes.
